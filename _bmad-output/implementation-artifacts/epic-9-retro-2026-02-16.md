### 2. Design Review Manquant Avant Implémentation (Insight Clé de Matt)
**Problème** : Story 9.2 avait **8 problèmes HIGH** trouvés en review finale, tous architecturaux:
- Missing shutdown timeout (10s timeout ajouté après review)
- Missing task handle storage (start_auto_persistence() refactoré)
- Missing shutdown channel (tokio::select! ajouté)
- Incomplete SIGTERM support (Unix signal handlers ajoutés)

**Impact** : 2-3 heures de corrections post-review pour Story 9.2  
**Root Cause** : Pas de design review avant implémentation. Elena a codé directement sans validation du design.  
**Solution appliquée** : Corrections en review, mais ça aurait dû être attrapé en design

**Constat Matt:** "Le point 2 (design review manquant) semble essentiel et il corrigera en partie le point 4 (nombre élevé de problèmes) également."

**Constat Charlie:** "Si on fait un design review **avant** le code, on attrape les problèmes architecturaux tôt - shutdown channels, race conditions, lock patterns."

### 3. Race Conditions Manquées par les Tests
**Problème** : Story 9.3 avait une **race condition TOCTOU** critique. Le code scannait les sessions expirées, puis les supprimait sans re-vérifier l'expiration.  
**Impact** : Bug trouvé en review, pas en tests initiaux  
**Root Cause** : Tests initiaux (13 tests) ne testaient pas les race conditions. `test_delete_session_toctou_protection()` ajouté seulement après review  

**Constat Dana:** "Ce qui m'inquiète c'est: pourquoi nos tests ne l'ont pas attrapé?"

**Constat Charlie:** "Les race conditions sont invisibles sans tests de concurrence spécifiques. On ne peut pas compter sur les tests 'normaux' pour les attraper."

### 4. Nombre Élevé de Problèmes en Review
**Problème** : 26 problèmes au total sur 3 stories (10 + 16 + plusieurs)  
**Impact** : Peer review rapide (30min) n'a pas suffi à réduire ce nombre  
**Root Cause** : Peer review attrape problèmes superficiels (magic numbers, logs) mais pas problèmes architecturaux (shutdown channels, race conditions)

**Constat Charlie:** "Pour attraper les problèmes architecturaux, il faut un design review **avant** l'implémentation, pas une peer review **après**."

## Apprentissages Clés

### 1. Design Review Avant Implémentation = Multiplicateur de Qualité (Matt's Key Insight)
**Apprentissage** : Design review de 30-45 minutes **avant** le code attrape les problèmes architecturaux (shutdown channels, race conditions, lock patterns) qui seraient trouvés en review finale autrement.

**Exemples concrets:**
- Story 9.2 avait 8 HIGH issues en review finale (shutdown timeout, task handle, shutdown channel, SIGTERM)
- Avec un design review avant le code, ces problèmes auraient été identifiés et résolus en 30-45 minutes de discussion
- Elena aurait implémenté avec un design validé, pas en devinant

**Impact attendu:** Réduction des HIGH issues de 8 → 2-3 par story. Économie de temps de correction.

**Application Epic 10:** Design review obligatoire pour toute story touchant à:
- Concurrence (locks, channels, async coordination)
- Intégration externe (APIs, bots, webhooks)
- Shutdown/graceful handling
- Patterns architecturaux nouveaux

### 2. Story Type 'Verification' Nécessite des Attentes Claires

- Elena confuse au début, a dû demander clarifications

**Application Epic 10:** Si story types non-standard utilisés, documenter explicitement:
- Quand utiliser ce story type

- Checklist des livrables attendus
- Exemples de bon/mauvais usage

### 3. Race Conditions Nécessitent des Tests Spécifiques
**Apprentissage** : Les tests "normaux" ne capturent pas les race conditions. Tests de concurrence spécifiques requis (TOCTOU scenarios, concurrent readers/writers).

**Exemples concrets:**

- Race condition trouvée en review par Charlie (expérience)
- `test_delete_session_toctou_protection()` ajouté après review

**Application Epic 10:** Dans design reviews, identifier explicitement:

- Planifier tests de concurrence dès le design
